OMG, What Even IS This?! 😱

Okay bestie, gather 'round because we're about to spill the ultimate tea on form-associated web components. You know how forms are like, the backbone of every website ever? Well, imagine if you could build your OWN custom form fields that are just as smart and accessible as the native ones — but with your own style. That's exactly what we're talking about!

"CSS encapsulation means the shadow DOM label, help text, input, and all focus/hover/error states can be styled with ordinary shadow-root CSS." — literally the dream 💅

The approach is built on three principles: place ARIA roles on inner shadow elements, use named slots for label and description content, and support cross-root element references via properties like ariaLabelledByElements. Let's get into it!

~ 💖 ~ 💖 ~ 💖 ~
Shadow DOM: Demystified! 🌑
What's Shadow DOM?

Think of it like your component's private bedroom. The outside world (Light DOM) can't mess with your styles or structure. Your component gets to be itself, totally encapsulated! 🚪


<my-input> host
<span id="label"> ← aria-labelledby points here
<input> ← the real star ⭐
<span id="description"> ← help text lives here
Two Types of Roles, Babe!

Native roles (textfield, checkbox): use an actual <input> element inside shadow DOM. Let the platform do its thing!

Explicit roles (progressbar, combobox): use a <div role="combobox"> inside shadow DOM. You're in control!

Combobox tip: keep the listbox in shadow DOM too, with same-root aria-controls and ariaActiveDescendantElement for the active option. 🎯

💡 Hot Tip: Always attach your shadow root with delegatesFocus: true so focus management just WORKS. No extra effort required!

IDREF Strategy: Your Labelling Glow-Up 💅
Two Ways to Label Your Component (Both Are Valid, OK?!)
✨ Method 1: Slots
<my-textfield>
  <span slot="label">
    Email
  </span>
  <span slot="description">
    Never shared.
  </span>
</my-textfield>

Project label and description content right into the component using named slots. Same-root ARIA wiring handles the rest automatically. Ideal for standalone fields! 🧼

DEFAULT ★
💫 Method 2: Properties
<my-textfield
  labelledby="col-header"
  describedby="note"
></my-textfield>

Pass IDs via properties and ariaLabelledByElements does the cross-root magic. Perfect for data grids reusing existing table headers, multi-section forms with shared headings, or inline-editing! 🎯

FLEXIBLE ✦
🤩 The BEST part? LabellingController merges BOTH sources automatically! Mix slots AND property references — like a describedby for a live error alert alongside a slotted default description. Yes, really!

Meet the Controller Queens 👑
🎀
Labelling
Controller

The OG. Watches slotchange events, shows/hides label and description spans, wires ARIA relationships, merges both labelling sources, provides debugInfo for dev inspection, and calls an onUpdate() callback after re-wiring. She does it ALL.

HEAD QUEEN
⚡
FieldAssociation
Controller

Wraps the ElementInternals object and provides setValue() for value reporting, default value tracking, and pass-through access to form properties. Call setValue(null) for disabled or unchecked fields to exclude them from FormData!

ESSENTIAL
🔮
ButtonAssociation
Controller

The interim queen! Handles keyboard activation, ARIA role defaults, focusability, and commandfor/command dispatch. Wire form submit/reset in a click listener alongside her. Just until native platform behaviors arrive!

INTERIM ✨
📋 How to use them: Instantiate your controller in the class body, call connect() in connectedCallback, and proxy your labelledby/describedby properties to the controller. Done! 💅

14 Rules to Live By 📖
The Component Authoring Commandments ✦

Memorize these, print them out, put them on your wall. Your component's reputation DEPENDS on it:

  • 1Shadow role NEVER on host — Put your ARIA role on inner shadow elements, not the custom element itself. Non-negotiable!
  • 2delegatesFocus: true always — Attach your shadow root with focus delegation. Your keyboard users will thank you.
  • 3Required IDs are sacred — Use exactly id="role", id="label", and id="description". Don't get creative here!
  • 4Labels hidden by default — Keep your label/description spans hidden; the controller reveals them when slot content exists.
  • 5Exact slot names only — Use "label" and "description". Not "Label", not "lbl". Exact match or it breaks!
  • 6LabellingController in class body — Instantiate it there, call connect() in connectedCallback. Order matters!
  • 7Proxy your properties — Forward labelledby and describedby to the controller. Let it do the heavy lifting!
  • 8NO ID refs from shadow to light — Never use ID attributes to cross the boundary. Use ariaLabelledByElements instead. Pinky promise! 🤙
  • 9ElementInternals wiring on the element class — Keep static formAssociated = true and attachInternals() on the class; controllers receive the internals object.
  • 10setValue(null) for disabled/unchecked fields — Call fieldAssoc.setValue(null) to exclude the field from FormData when disabled or unchecked. Don't forget!
  • 11Wire form submit/reset in a click listener — Buttons need manual click handlers for submit/reset wiring. Use ButtonAssociationController for activation alongside it.
  • 12Support both labeling modes & merge them — Your component should handle slotted content AND external property-based refs, merging them when both are present.
  • 13Combobox: keep listbox in shadow DOM — Use same-root aria-controls and ariaActiveDescendantElement for the active option. No cross-root drama needed!
  • 14Test with screen readers — axe-core has known gaps with ElementInternals patterns. Screen reader testing is the true authority!

axe-core Drama: The Full Receipts 🧾
False Positives AND False Negatives — Sis Has Range 😤
🚨 False Positives (axe yells when it shouldn't)
  • label rule — inspects the host but doesn't traverse to shadow role elements
  • aria-required-children — doesn't follow slotted content into light DOM
🔇 False Negatives (axe stays quiet when it shouldn't)
  • Unlabeled inner elements when cross-root references fail silently
  • Stale element references when targets are removed from the DOM
Screen reader testing is the TRUE authority. axe-core is a helper, not the final word! Revisit quarterly as Deque ships fixes. 📅

Strategy: exclude affected rules per component with rationale comments and upstream issue links.

Future Looks: Next Season's Forecast 🔮
What's Coming Down the Runway...
WHATWG STAGE 1
HTMLSubmitButton­Behavior

Unlocks all of this just by passing a behavior to attachInternals():

  • Implicit form submission (Enter in a text field)
  • The :default CSS pseudo-class
  • Native keyboard activation (Enter, Space)
  • Mirrored properties identical to HTMLButtonElement

All three browsers have filed standards positions. Chromium/Edge is implementing! 🎉

FLAG-GATED
referenceTarget

Lets the shadow root declare an inner element as the canonical IDREF target. Once shipped unflagged:

  • Consumers use standard aria-labelledby on the host
  • No labelledby/describedby component properties needed
  • No cross-root element reference JS wiring

The dream! ✨ LabellingController still needed for slotted content.

🌟 Your Migration Glow-Up Path:

1️⃣ Now: Use controllers as interim patterns — solid and production-ready!
2️⃣ When referenceTarget ships: Consumers switch to standard aria-labelledby. No component changes needed!
3️⃣ When HTMLSubmitButtonBehavior ships: Replace ButtonAssociationController with attachInternals({ behaviors: [new HTMLSubmitButtonBehavior()] }) and remove manual click handlers. One surgery!

How To Use It: Real Code Examples! 💻
Copy-Paste These Into Your Life ✂️

Three flavors — pick what fits your layout and go!

✨ Slotted Content CLEANEST

You own all the label text inside the component. Perfect for standard forms!

<my-textfield>
  <span slot="label">Email</span>
  <span slot="description">Never shared.</span>
</my-textfield>
💫 Property-Based DATA GRIDS

Point to existing labels on the page — no duplicate text!

<my-textfield labelledby="col-header" describedby="note"></my-textfield>
🔥 Hybrid Mode MOST FLEXIBLE

Mix slotted content with an external error reference. Both merge together!

<my-textfield describedby="error">
  <span slot="label">Email</span>
  <span slot="description">Default help text.</span>
</my-textfield>
<p id="error" role="alert">This field is required.</p>
