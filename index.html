Form fields strategy demos

Interactive PoCs exploring how label and description connect across Shadow DOM boundaries for textfield, checkbox, progress bar, and combobox components. In all demos the ARIA role lives on an inner shadow DOM element, not the custom element host.

Project README · Open in StackBlitz

Shadow DOM label and description

Label and description live inside the shadow DOM with <slot name="label"> and <slot name="description"> for consumer-provided text. The role element is wired via aria-labelledby and aria-describedby using same-root IDs — no cross-root ID references needed. The combobox uses Element.ariaActiveDescendantElement for its active option.

  • Shadow role — shadow DOM label and description

Light DOM label and description siblings

Label and description are plain light DOM siblings of the custom element — no slots, no shadow markup for them. Each component exposes labelledby and describedby properties that accept element ID strings. Setting them resolves the referenced elements in the document and updates Element.ariaLabelledByElements and Element.ariaDescribedByElements on the inner role element, crossing the shadow boundary via the element reference API.

  • Shadow role — light DOM label and description siblings

Hybrid — slotted with light DOM fallback

Each component has shadow DOM label and description spans with named slots. When slot content is present, ariaLabelledByElements points at the shadow span (same-root element reference). When slots are empty, it falls back to ariaLabelledByElements pointing at light DOM siblings via the labelledby / describedby properties. A toggle button on the demo page switches between the two modes live so you can inspect the difference.

  • Shadow role — hybrid shadow/light label and description
